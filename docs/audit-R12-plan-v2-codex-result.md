# New problems in v2

1. [R12.3 split] None identified.
   Source requires this split: sync compound calls `Order::execute`, `modify`, and `cancel` (`src/compound_order.rs:405-414`, `:421-433`), and those methods call broker-copy hooks before broker operations (`src/order.rs:568-570`, `:628-629`, `:782-789`). `AsyncBroker` already has async hook siblings (`src/async_broker.rs:75-83`). v2 says R12.3a adds async order siblings with the same attrib-copy params before R12.3b uses them (`docs/R12-async-complete-plan.md:231-253`, `:280-290`).

2. [R12.1 lossy adapter] None identified.
   The lossy `AsyncBroker` adapter is sufficient for `AsyncCompoundOrder` because sync compound/order already use only the broker's `Option<String>`/unit lifecycle surface, not `VirtualBroker::BrokerReply` (`src/order.rs:609-614`, `:688-689`, `:789`; `src/virtual_broker.rs:22-30`, `:163-168`). v2 keeps rich `BrokerReply` only on inherent `AsyncVirtualBroker` methods for parity tests and uses `Arc<dyn AsyncBroker>` for compound execution (`docs/R12-async-complete-plan.md:118-146`, `:283-290`).

3. [R12.5 publish order] No new version-pin mismatch identified.
   v2's order is now `polymarket-kernel` publish/docs.rs, then `omsrs` publish/docs.rs, then pbot switches both deps from path to registry versions and runs its 305-test suite (`docs/R12-async-complete-plan.md:382-416`).
